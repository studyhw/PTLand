## 项目概述

**PTLand** 是一个双机联动的电池自动化测试系统，用于在长时间充放电循环中，验证笔记本电脑在断电、关机、重启等场景下的稳定性与可靠性。

### 主要要解决的问题

- **无人值守的长时间电池充放电循环测试**：
  自动控制 PDU 插座的通断电，在多圈循环中反复模拟“拔市电 → 电池放电 → 电量耗尽自动关机 → 再上电重启 → 重新充满电”的完整流程，避免人工反复插拔电源和盯电量。
- **断电重启后测试无法连续的问题**：
  通过在客户端持久化当前状态与圈数，并配置开机自启，实现设备因电池耗尽关机后重新上电、自动启动客户端并继续按上次进度续跑测试。
- **难以同时覆盖高负载与电源异常场景的问题**：
  在放电与充电阶段自动启动/关闭 BurnInTest 和 ECTool，对系统施加负载并记录数据，在“高负载 + 断电 + 重启 + 充电”的真实组合场景下验证整机稳定性和可靠性。
- **双机协同与电源控制难以联动的问题**：
  通过 RPC 让客户端与服务端通信并周期性发送心跳，服务端结合心跳与 PDU 控制实现看门狗逻辑，当检测到客户端长时间离线时自动判断为关机状态并在合适时机重新上电。

**控制端 (`control_server.py`)**：运行在连接 PDU 的控制电脑上，通过 SNMP 控制插座电源通断，并通过 RPC 接收客户端指令和心跳。

**客户端 (`client.py`)**：运行在被测笔记本上，负责执行充放电循环逻辑、启动/关闭 BurnInTest 和 ECTool、维护测试状态，并在

* col1col2col3

断电重启后自动恢复测试。

运行环境：

- Windows 10 / 11
- Python 3.8
- 依赖库见 `requirements.txt`

## 项目结构

项目根目录主要文件如下：

- `control_server.py`控制端主程序，负责：

  - 启动 `rpyc.ThreadedServer`，监听端口 `18861`
  - 初始化并管理与 PDU 的 SNMP 通信
  - 处理来自客户端的 RPC 请求（上电、断电、心跳等）
  - 实现看门狗逻辑（在客户端长时间掉线后自动上电）
- `client.py`客户端主程序，负责：

  - 提供 Tkinter GUI 界面，配置测试参数与路径
  - 管理电池充放电状态机
  - 启动/关闭 BurnInTest、ECTool 等测试工具
  - 通过 RPC 与控制端通信，并定期发送心跳
  - 维护本地配置与当前测试进度
- `pdusnmp.py`PDU 相关的 SNMP 封装模块，供 `control_server.py` 调用，用于：

  - 初始化 SNMP 引擎
  - 读取 PDU 设备信息
  - 控制指定插孔的上电/断电
- `requirements.txt`项目依赖列表（包括 `rpyc`、`psutil`、`tkinter`、`pysnmp==4.4.1`、`pyasn1==0.4.8` 等）。
- `icon.ico`客户端 GUI 使用的图标文件（可选，如存在则自动加载）。
- `test_config.json`（运行过程中生成）用于持久化客户端配置与测试进度，包括：

  - Server IP、ECTool 路径、BurnInTest 路径
  - 当前测试圈数、当前状态（IDLE/DISCHARGING/WAITING_S5/CHARGING 等）
- `plan.md`
  项目需求与设计说明初稿（用于开发指导）。

## 功能说明

### 1. 控制端 (`control_server.py`)

- **应用标题**：控制台标题为 `PTLand_Server`。
- **RPC 服务**：
  - 使用 `rpyc.ThreadedServer`，监听 `0.0.0.0:18861`
  - 向客户端暴露上电、断电、心跳等 RPC 接口
- **PDU 控制**：
  - 使用 `pdusnmp` 模块
  - 当前硬编码：PDU IP 为 `192.168.0.163`，插孔 Index 为 `1`
  - 启动时初始化 SNMP，引导出设备名并打印
- **看门狗逻辑**：
  - 接收客户端心跳，记录最近一次心跳时间
  - 当客户端请求“断电”后，启用看门狗监控
  - 若连续 30 秒未收到心跳（视为客户端因电池耗尽进入 S5）
    - 再等待 30 秒（确保彻底断电）
    - 然后通过 PDU 执行上电操作，唤醒设备
- **日志系统**：
  - 使用 `logging` 模块
  - 日志格式含时间戳 `[HH:MM:SS]`、日志级别与消息内容

### 2. 客户端 (`client.py`)

- **GUI 界面**：

  - 标题为 `PTLand`
  - 使用 Tkinter 构建可滚动的主界面（`Canvas` + `Scrollbar` + `Frame`）
  - 窗口缩小时内容可滚动，放大时输入框和日志区域自动扩展
  - 主要控件包括：
    - Server IP 输入框
    - 总圈数设置
    - ECTool 路径
    - BurnInTest 路径
    - 开始 / 停止 / 重置 按钮
    - 日志文本框（显示当前状态与历史记录）
  - 如存在 `icon.ico`，启动时加载为窗口图标
- **状态机逻辑**（核心测试流程）：

  1. **IDLE（空闲）**

     - 启动时读取当前电池电量与历史配置
     - 若电量 > 99%，进入 `DISCHARGING`（放电）
     - 否则通过 RPC 请求控制端上电进行充电
  2. **DISCHARGING（放电）**

     - 清理残留进程：强制结束 `bit.exe`、BurnInTest、ECTool 等
     - 启动 ECTool（`cwd` 设置为其所在目录）
     - RPC 请求控制端断电（由电池供电）
     - 使用命令行启动 BurnInTest：`/r /D 0`（自动运行、无限时长）
     - 持续监控电池电量，当电量 ≤ 3%：
       - 关闭所有测试软件
       - 状态切换为 `WAITING_S5`
  3. **WAITING_S5**

     - 等待 Windows 自动休眠/关机
     - 预先将下一状态保存为 `CHARGING` 并写入 `test_config.json`
     - 设备关机后由 PDU 看门狗在合适时机重新上电
  4. **CHARGING（充电）**

     - 系统重启后，客户端读取配置，发现状态为 `CHARGING`
     - 启动后首先通过 RPC 再次请求上电（双重保险）
     - 启动 ECTool 记录充电过程
     - 当电量充至 100%：
       - 当前圈数 +1 并保存
       - 若未达到总圈数，返回 `DISCHARGING` 继续下一轮
       - 达到总圈数后进入完成状态，停止测试
- **持久化与自恢复**：

  - 所有关键配置与状态写入 `test_config.json`
  - 任意时刻断电，只要下次重启后仍能启动 `client.py`，即可根据状态自动恢复测试流程
- **开机自启**：

  - 测试开始时，将自身路径写入 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`
  - 测试停止或全部圈数完成后，自动删除对应自启项
- **网络与心跳**：

  - 在后台线程中维护与控制端的连接，自动断线重连
  - 周期性向控制端发送心跳，供看门狗逻辑使用

## 使用说明（操作手册）

### 一、环境准备

1. 安装 Python 3.8（64 位），并确保已添加到系统环境变量 `PATH`。
2. 在项目根目录执行依赖安装：

```bash
pip install -r requirements.txt
```

3. 确保：
   - 控制端电脑能通过网络访问 PDU（默认 IP：`192.168.0.163`）
   - 被测笔记本与控制端电脑在同一网络或可互通
   - 已在本地安装 BurnInTest 与 ECTool，并知道其可执行文件路径

### 二、启动控制端

1. 在控制机上打开命令行，进入项目目录。
2. 运行：

```bash
python control_server.py
```

3. 控制台标题应显示为 `PTLand_Server`，启动日志中会打印 PDU 设备名，以及 RPC 服务监听端口等信息。
4. 确认无报错后，保持控制端程序运行，不要关闭窗口。

### 三、配置并启动客户端

1. 在被测笔记本上打开命令行，进入项目目录。
2. 运行：

```bash
python client.py
```

3. 在弹出的 GUI 窗口中完成以下设置：

   - **Server IP**：填入控制端电脑的 IP 地址
   - **总圈数**：设定本次需要执行的充放电循环次数
   - **ECTool 路径**：选择或输入 ECTool 的可执行文件路径
   - **BurnInTest 路径**：选择或输入 BurnInTest 的可执行文件路径
4. 点击 **开始** 按钮：

   - 程序会保存配置到 `test_config.json`
   - 写入 Windows 注册表，自身设为开机自启
   - 根据当前电池电量进入对应状态（IDLE/DISCHARGING/CHARGING）
5. 在测试过程中：

   - 日志窗口会持续输出当前状态、圈数进度、电池信息等
   - 如需中止测试，可点击 **停止** 按钮
   - 如需完全重置（清空进度与配置），可使用 **重置** 按钮（如有该功能）

### 四、测试流程示意

1. 设备满电 → 断市电 → 电池放电 → 电量耗尽 → 自动关机
2. 控制端监测到心跳丢失超过阈值 → 延迟后自动上电
3. 设备从市电启动 → 自动运行客户端程序（注册表自启）→ 进入充电状态
4. 满电后圈数 +1，重复步骤 1–3，直到达到设定总圈数。

## 注意事项

- **安全性**：

  - 确保 PDU 接线正确，并确认仅控制被测设备的电源插孔
  - 测试过程中设备会频繁断电重启，请确保数据无关或已备份
- **可靠性**：

  - 建议在首轮测试前，先以 1–2 圈进行试跑，确认流程配置正确
  - 如发现 ECTool 或 BurnInTest 未正常启动，请检查路径与权限（可尝试以管理员方式运行）
- **扩展与修改**：

  - 如需更改 PDU IP 或插孔 Index，建议在 `control_server.py` 中抽出为配置项
  - 可在 `client.py` 中扩展更多测试状态或增加其他测试工具的集成

### 五、推荐的落地测试步骤（简版 Checklist）

1. **硬件与网络**
   - 控制机、被测机、PDU 接线正确，被测机电源适配器插在 PDU 指定插孔上。
   - 控制机能 `ping` 通 PDU IP，被测机能 `ping` 通控制机 IP。
2. **控制端检查**
   - 在 `control_server.py` 中把 PDU IP、插孔 Index 改成现场实际值。
   - 在控制机上安装依赖并运行 `python control_server.py`，确认日志里能正确打印 PDU 信息、无报错。
3. **客户端试跑（建议先 1–2 圈）**
   - 被测机安装 BurnInTest、ECTool，并确认能手动双击启动。
   - 运行 `python client.py`，在 GUI 中配置 Server IP、总圈数（1–2 圈）、ECTool 路径、BurnInTest 路径，点击“开始”。
   - 观察客户端日志与控制端日志，确认能正常连接、PDU 能按阶段断电/上电、电池电量变化和圈数递增正常。
4. **验证自动关机与自动续跑**
   - 放电阶段让电量自然耗尽到自动关机，查看控制端日志：心跳丢失一段时间后自动上电。
   - 设备重启后，客户端应自动启动并从上次状态继续，而不是从 0 圈开始。
5. **长时间正式测试前**
   - 确认 `test_config.json` 中 IP、路径、圈数配置正确，控制端 PDU 参数与现场保持一致。
   - 将总圈数改为目标值（如 50/100），在环境稳定的前提下开始长时间无人值守测试。

### 六、什么时候算测试结束？

- 当日志中显示 **“已完成设定总圈数”** 或类似提示，并且状态不再切换时，说明：
  - 所有设定圈数的充放电循环已经跑完
  - 程序会自动移除开机自启（取决于实现）
  - 你可以手动关闭客户端窗口和控制端窗口

### 七、常见小问题

- **客户端连不上服务端**

  - 检查 Server IP 是否填错
  - 检查两台机器是否在同一网段或网络互通
  - 确认控制端窗口还在运行，没有被关掉
- **BurnInTest / ECTool 没启动起来**

  - 先手动双击验证路径是否正确
  - 尝试用管理员权限运行测试脚本
  - 检查杀毒软件或安全策略是否拦截
- **电量不变化或圈数不增加**

  - 检查 PDU 是否真的断电 / 上电
  - 检查客户端日志中是否有状态一直卡在某一阶段
